01Overview

This document records the full process of integrating Microsoft Sysmon with Splunk Enterprise in a Windows Server 2025 homelab environment. It covers initial evidence collection, diagnosis of sourcetype misassignment, root cause identification in a Splunk Technology Add-on (TA), and the final resolution.

The troubleshooting methodology demonstrated here reflects real-world Splunk administration practice.

02Initial Evidence & Problem Statement

Initial Splunk searches confirmed that Sysmon was generating and ingesting events, but the sourcetype was incorrectly assigned. The following was observed:

  • ⚠️ index=wineventlog returned 45,927 events with 4 sourcetypes: WinEventLog:Security, WinEventLog:System, WinEventLog:Application, xmlwineventlog — no dedicated Sysmon sourcetype visible.
  • ⚠️ Searching by source returned 37,972 Sysmon events confirming data was present.
  • ❌ Searching by sourcetype="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational" returned 0 events.
  • ✅ The SysmonProcessMonitor dashboard was populated, confirming Sysmon telemetry was reaching Splunk.
🔴
Core Problem Sysmon events were landing under the generic xmlwineventlog sourcetype instead of their own dedicated sourcetype, making targeted detection and dashboard queries unreliable.

03Diagnostic Process

1
Confirming Sysmon Service Status

Verified Sysmon was running and actively logging to the Windows Event Log:

PowerShellGet-Service Sysmon64
Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational" -MaxEvents 5
✅
ResultSysmon was active, generating hundreds of Event ID 1 (Process Create) events per minute. The problem was therefore in Splunk configuration, not Sysmon itself.
2
Sourcetype Audit

Ran a sourcetype breakdown across all wineventlog events to identify what sourcetypes existed:

SPLindex=wineventlog | stats count by sourcetype
⚠
ResultOnly 4 sourcetypes present — xmlwineventlog, WinEventLog:Security, WinEventLog:System, WinEventLog:Application. No Sysmon-specific sourcetype.
3
Confirming Sysmon Data Presence by Source
SPLindex=wineventlog source="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational"
| stats count by sourcetype
⚠
Result86,558 events found — all stored as xmlwineventlog. This confirmed the data existed but was mislabelled.
4
inputs.conf Investigation

Examined the Splunk inputs.conf to understand how Sysmon events were being collected:

PowerShellGet-Content "C:\Program Files\Splunk\etc\system\local\inputs.conf"
⚠
Findings The Sysmon stanza had no explicit sourcetype directive, a whitelist limiting collection to Event IDs 8, 11, 12, 13, 22, and a blacklist that initially included Event ID 1 — conflicting with the whitelist. Both issues were corrected.
5
btool Configuration Verification

Used Splunk's btool utility to verify the merged configuration Splunk was actually applying:

CMD.\splunk.exe btool inputs list "WinEventLog://Microsoft-Windows-Sysmon/Operational" --debug
🔴
Key Discoverybtool confirmed sourcetype was set correctly in inputs.conf — yet it still appeared as xmlwineventlog in search results. This indicated an override at a higher precedence level.

04Root Cause Discovery

Examined the Splunk Technology Add-on for Sysmon default configuration:

PowerShellGet-Content "C:\Program Files\Splunk\etc\apps\Splunk_TA_microsoft_sysmon\default\props.conf"
Root Cause Identified

The TA's default/props.conf contained a rename directive described as a backward compatibility measure:

props.conf[XmlWinEventLog:Microsoft-Windows-Sysmon/Operational]
rename = xmlwineventlog
🔴
Why This Overrode Everything The rename directive operates at the sourcetype level and has the highest precedence in Splunk's configuration merge order. It cannot be overridden by inputs.conf sourcetype declarations — regardless of their location.

Splunk Configuration Merge Order

PrecedenceLocationNotes
1 (lowest)System defaultSplunk built-ins
2App default$APP/default/
3App local$APP/local/
4 (highest)System local$SPLUNK/etc/system/local/
⚠ Overrideprops.conf renameOperates above all inputs.conf declarations

05Resolution

ℹ
Industry Best Practice Technology Add-ons should never have their default/ directories modified directly. Create a local/ directory override within the TA — it survives TA updates and is documented in Splunk's official administration guide.

Step 1 — Create TA Local Override

1
Create local directory in the TA
PowerShellNew-Item -ItemType Directory "C:\Program Files\Splunk\etc\apps\Splunk_TA_microsoft_sysmon\local\"
2
Create props.conf override — cancel the rename directive

Create props.conf in the local directory with the following content:

props.conf[source::XmlWinEventLog:Microsoft-Windows-Sysmon/Operational]
sourcetype = XmlWinEventLog:Microsoft-Windows-Sysmon/Operational

[XmlWinEventLog:Microsoft-Windows-Sysmon/Operational]
rename =
ℹ
The empty rename = directive cancels the TA's default rename. The source:: stanza ensures new events are tagged correctly at ingestion time.

Step 2 — Finalise inputs.conf

3
Final inputs.conf Sysmon stanza
inputs.conf[WinEventLog://Microsoft-Windows-Sysmon/Operational]
sourcetype    = XmlWinEventLog:Microsoft-Windows-Sysmon/Operational
index         = wineventlog
disabled      = false
current_only  = 1
renderXml     = true
whitelist     = 1,8,11,12,13,22
blacklist     = 3,10
⚠
Secondary Issue FixedEvent ID 1 had been on both the whitelist and blacklist. Splunk's blacklist takes precedence, so Process Create events were silently dropped. Removing it from the blacklist restored collection.

Step 3 — Restart & Verify

4
Restart Splunk and confirm fix
CMDcd "C:\Program Files\Splunk\bin"
.\splunk.exe restart
SPL — Verificationindex=wineventlog source="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational"
| stats count by sourcetype
✅
ResultSourcetype now correctly displays as XmlWinEventLog:Microsoft-Windows-Sysmon/Operational across all Sysmon events.

07Key Lessons & Skills Demonstrated

🔧
Splunk Configuration Precedence
Splunk merges config from multiple layers in strict order: system default → app default → app local → system local. The rename directive in props.conf operates at the sourcetype level and overrides inputs.conf declarations regardless of file location.
📦
TA Customisation Best Practice
Technology Add-ons should never have their default/ directories modified directly. Creating a local/ directory with overrides is the correct approach — it survives TA updates and is the method documented in Splunk's official administration guide.
🔍
Systematic Troubleshooting
The investigation followed a logical sequence: confirm data exists → identify where it lands → verify configuration at each layer → use btool to inspect merged config → trace the override to its source. This methodology applies to any Splunk ingestion problem.
⚠️
Whitelist/Blacklist Interaction
Splunk's blacklist takes precedence over whitelist for WinEventLog inputs. Event ID 1 had been placed on both lists, resulting in Process Create events being silently dropped. Removing it from the blacklist restored collection.
